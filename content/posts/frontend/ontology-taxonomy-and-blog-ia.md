# 택소노미에서 온톨로지로

[문서형 블로그 IA를 먼저 설계한 이유](/posts/minimal-blog-ia)에서 "분류 체계가 느슨할수록 다시 찾기 어려워진다"고 적었다. 그때는 카테고리와 태그 이야기에 머물렀는데, [온톨로지·토폴로지·택소노미 정리](/posts/structure-terms-ontology-topology-taxonomy)를 쓰고 나니 같은 문제를 더 또렷한 언어로 볼 수 있게 됐다.

블로그의 정보 구조(IA)는 사실 그 세 가지 "구조"를 **동시에** 쓰고 있다. 이 글에서는 이 블로그를 예로 들어, 각 구조가 어디에 쓰이고 어디서 한계에 부딪히는지 정리한다.

---

## 1단계 — 카테고리는 택소노미다

이 블로그의 카테고리는 `Frontend`, `Network`, `React / Next.js`처럼 **하나의 글이 하나의 카테고리**에 속한다. 위에서 아래로 줄을 세우는 전형적인 **택소노미(계층 분류)**다.

```
Posts
├─ Frontend
├─ Network
├─ React / Next.js
└─ ...
```

장점은 명확하다. 직관적이고, 폴더처럼 머릿속에 잘 들어온다. 그런데 글이 쌓이면 택소노미의 한계가 드러난다.

- "SSE를 Next.js 프록시로 중계한" 글은 **Network**인가 **React / Next.js**인가?
- 한 카테고리에만 넣어야 하니, 다른 길로 찾아온 독자는 놓친다.

**한 부모만 허용**하는 택소노미의 본질적 약점이다.

---

## 2단계 — 태그와 관련 글은 토폴로지다

그래서 태그가 필요하다. 한 글에 `SSE`, `Next.js`, `Network`를 동시에 붙이면, 글은 더 이상 트리의 한 잎이 아니라 **여러 글과 옆으로 이어진 노드**가 된다. 카테고리가 위아래라면, 태그로 엮인 그물은 **토폴로지(연결 구조)**다.

이 블로그의 "관련 글" 추천도 같은 토폴로지 위에서 동작한다. 실제 코드(`getRelatedPosts`)는 **공유 태그 수**와 **같은 카테고리 여부**로 점수를 매겨 가까운 글을 고른다.

```
공유 태그 1개당 +1, 같은 카테고리면 +2 → 점수 높은 순으로 관련 글
```

즉 관련 글 그물은 "누가 누구와 태그를 공유하나"라는 **무방향 연결 지도**다. 카테고리가 못 잡는 가로질러 가는 길을 태그가 메운다.

> 택소노미(카테고리)로 큰 줄기를 세우고, 토폴로지(태그·관련 글)로 가로 연결을 깐다 — 대부분의 블로그가 여기까지 온다.

---

## 3단계 — 의미를 붙이면 온톨로지에 가까워진다

여기서 한 걸음 더 갈 수 있다. 지금의 "관련 글"은 **연결의 종류를 구분하지 않는다.** A와 B가 이어졌다는 건 알지만, *어떻게* 이어졌는지는 모른다.

그런데 실제 글 사이엔 종류가 다른 관계가 있다.

- [HTTP 기초](/posts/http-basics)는 [TCP/UDP/QUIC](/posts/tcp-udp-quic)의 **선행 지식이다**(prerequisite)
- [SSE 실전](/posts/sse-shared-worker-in-practice)은 [SSE 개념](/posts/server-sent-events)의 **응용편이다**(applies)
- [네트워크 토폴로지](/posts/network-topology)는 이 글과 **개념을 공유한다**(see-also)

이렇게 **관계에 이름을 붙이면** 그게 바로 **온톨로지**다. 단순한 "관련 글"이 아니라, "이 글을 읽기 전에 읽어야 할 글", "이 글의 다음 단계"를 구분해 보여줄 수 있다. 학습 경로(learning path)가 데이터에서 자동으로 나온다.

```
http-basics ──prerequisite──▶ tcp-udp-quic
server-sent-events ──applied-by──▶ sse-shared-worker-in-practice
```

지금 당장 구현하지 않더라도, **프론트매터에 관계 유형을 한 줄 더 적는 것**만으로 토폴로지를 온톨로지 쪽으로 끌어올릴 수 있다. 예:

```
related:
  - slug: tcp-udp-quic
    relation: prerequisite
```

---

## 정리 — 같은 글, 세 겹의 구조

| 층 | 이 블로그의 구현 | 한계 |
|---|---|---|
| 택소노미 | 카테고리 | 한 글이 한 곳에만 |
| 토폴로지 | 태그·관련 글 | 연결의 *종류*를 모름 |
| 온톨로지 | (아직) 관계 유형 메타데이터 | 작성 비용이 듦 |

세 가지는 경쟁하지 않는다. **택소노미로 줄기를 세우고, 토폴로지로 가로 연결을 깔고, 필요한 곳에만 온톨로지로 의미를 더하는** 순서가 자연스럽다.

[IA 설계 글](/posts/minimal-blog-ia)에서 "탐색 구조를 먼저 잡으라"고 했던 말은, 결국 이 세 겹 중 **어디까지 갈지를 미리 정하라**는 뜻이기도 했다. 지금 이 블로그는 2단계(토폴로지)에 있고, 3단계는 글이 더 쌓였을 때의 숙제로 남겨 둔다.
